# High Concurrency Project Roadmap

## 1. 项目背景与目标

本项目是一个 **长期练习型工程项目**，目标不是交付业务系统，而是通过持续实践，系统性提升以下能力：

- 高并发、高吞吐系统设计能力
- Java / JVM 性能分析与调优能力
- Kafka / Redis 等中间件的工程化使用能力
- 微服务拆分、部署、监控、灰度发布能力
- 从“实现功能”到“定义问题并解决问题”的工程思维转变

### 核心原则

- **实践优先于理论**
- **允许失败，但必须复盘**
- **所有结论必须来自数据和现象**
- **系统逐步演进，不追求一步到位**

---

## 2. 系统总体定位

### 系统名称（内部使用）

> High Throughput Event Processing Platform  
> 高并发实时事件处理系统

### 抽象问题定义（一句话）

在高并发写入场景下，系统能够：
- 稳定接收大量事件数据
- 通过消息队列进行削峰解耦
- 对事件进行异步处理与聚合
- 使用缓存与数据库支撑低延迟查询
- 在压力和故障下具备可观测性与可恢复性

## 4. 技术栈约定（长期）

- JDK: 17
- Spring Boot: 3.x
- 构建工具: Maven
- 消息队列: Kafka
- 缓存: Redis
- 数据库: MySQL
- 容器化: Docker / Docker Compose
- 监控: Micrometer + Prometheus + Grafana
- 压测工具: JMeter / wrk / Gatling
- CI/CD: Shell / GitHub Actions（后期）

## 5. 阶段划分与学习路线

---

### 阶段一：系统基座 & 并发入门（Weeks 1–4）

#### 核心目标
- 建立完整可运行的本地系统环境
- 理解 JVM 在高并发下的基本行为
- 掌握 Kafka 作为削峰工具的基本用法
- 初步建立监控与压测意识

#### 关键实践内容
- Docker Compose 管理中间件
- Ingest Service 高并发接入
- JVM 参数配置与 GC 行为观察
- 基础压测与指标采集

#### 输出物
- docker-compose v1
- Ingest Service
- 第一次压测报告
- Week 1–4 总结文档

---

### 阶段二：高吞吐 & 数据处理（Weeks 5–12）

#### 核心目标
- 深入理解 Kafka 吞吐、分区、消费模型
- 实现异步处理与实时聚合
- 掌握 Redis 缓存热点问题

#### 关键实践内容
- Stream Processor 服务
- Kafka Consumer 并发模型
- 幂等、重试、失败补偿设计
- Redis Key 设计与内存控制

#### 输出物
- Stream Processor 可运行版本
- Kafka Lag 监控
- Redis 命中率与热点分析
- 多轮压测与调优记录

---

### 阶段三：微服务拆分与系统稳定性（Months 4–6）

#### 核心目标
- 明确服务边界
- 提升系统稳定性和容错能力
- 从“跑起来”走向“跑得稳”

#### 关键实践内容
- Query Service
- 服务限流、降级、兜底
- 系统容量评估
- 故障模拟与恢复演练

#### 输出物
- 系统架构设计文档
- 稳定性方案说明
- 故障演练复盘文档

---

### 阶段四：部署、灰度与 CI/CD（Months 6–9）

#### 核心目标
- 掌握工程级交付能力
- 支持系统安全发布和快速回滚

#### 关键实践内容
- Docker Image 标准化
- CI 自动构建
- 灰度发布策略（接口级 / 消费者级）
- 回滚机制验证

#### 输出物
- CI/CD 脚本
- 灰度发布方案文档
- 发布事故模拟与复盘

---

### 阶段五：进阶扩展（Months 9–12，可选）

#### 核心目标
- 接触集群与分布式复杂性
- 提升系统视角和架构能力

#### 关键实践内容
- 多 Kafka Broker
- Redis Cluster
- 多节点部署
- 网络与节点故障模拟

---

## 6. 统一实践与评估标准

每个阶段、每个 Week，至少输出：

- 一篇实践文档
- 一次压测结果
- 一张监控图
- 一个明确的问题与结论
- 下一步优化方向

### 核心评估问题

- 系统瓶颈在哪里？
- 为什么会在这里？
- 如何通过工程手段解决？
- 有哪些 trade-off？

---

## 7. 协作与自我约束方式

- 所有代码、配置、文档必须进入 Git
- 所有“拍脑袋结论”必须被数据验证
- 不追求一次做对，追求持续变强
- 本项目以 **工程成长** 为唯一成功标准
